## Why

ユーザーが自分だけのノートにアクセスするためには認証機能が必要。Step 1 で環境構築が完了し、次のステップとしてログイン・ログアウト機構を実装することで、ユーザー別のデータ管理が可能になる。

## What Changes

- バックエンドに認証 API を追加
  - `POST /api/login`: ユーザー名・パスワードでログイン、セッショントークンを Cookie で返却
  - `POST /api/logout`: セッションを破棄し Cookie を無効化
  - `GET /api/me`: 現在のログイン状態を確認
- 認証ミドルウェアを追加（以降のメッセージ系 API で使用）
- フロントエンドにログイン画面を追加
- フロントエンドに認証状態管理を追加（起動時の認証確認、ログアウトボタン）

## Capabilities

### New Capabilities

- `session-auth`: Cookie ベースのセッション認証。ログイン、ログアウト、セッション検証、認証ミドルウェアを含む
- `login-page`: ユーザー名・パスワード入力によるログイン画面

### Modified Capabilities

（なし - 新規機能のみ）

## Impact

- **バックエンド**:
  - `backend/` に認証関連のハンドラーとミドルウェアを追加
  - `bcrypt` パッケージへの依存追加
  - `sessions` テーブルへの読み書き
- **フロントエンド**:
  - `frontend/src/` にログイン画面コンポーネントを追加
  - ルーティングまたは条件分岐による画面切り替えロジック
  - 認証状態の管理（Context または state）
- **API**:
  - 3つの新規エンドポイント（`/api/login`, `/api/logout`, `/api/me`）
